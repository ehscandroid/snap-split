A portable reference for a Claude instance with no access to this repository — component inventory, design tokens, and rough layout mocks, pulled from the real source rather than invented.

1 · Overview

What it is. The BildWert.com "mother repo" — a component library and showcase app. It hosts the canonical Button and FormElements families later copied into sibling project repos, plus a working domain demo (SDS / chemical safety-data-sheet management: SDS records, Tenants, Packages) built on a resizable three-panel "split view" shell.

2 · Design tokens

Theme (light/dark)

Class-based via .dark on <html>, toggled by ThemeContext (useTheme), persisted to localStorage['app-theme'], defaults to prefers-color-scheme. Tailwind v4 custom variant: @custom-variant dark (&:where(.dark, .dark *));

Accent color

AccentContext (useAccent) — 7 named colors, persisted to localStorage['app-accent'], applied as --accent / --accent-hover CSS custom properties on document.documentElement. Consumed via inline style or color-mix(in srgb, var(--accent) N%, transparent) — never a Tailwind color class, so it can react at runtime.

3 · Layout & shell

Three-panel "split view" (splitview/Layout.tsx + splitview/Panel.tsx): outer flex row, h-screen w-screen, p-4 gap-4. Panel widths persist per-panel to localStorage. Each panel is a rounded-2xl bordered card — sticky header (shadow appears on scroll), scrollable body, optional sticky footer, draggable right-edge resize handle. Below collapseWidth (default 40px) a panel auto-collapses to icon-only; can also be toggled manually when closable.

5 · Data model

From src/types.ts:

interface SdsItem {
  id: number
  name: string
  casNumber: string
  hazardClass: string
  status: number          // → STATUS_MAP code
  manufacturer: string
  signalWord: string       // 'Danger' | 'Warning' | 'None'
  storageClass: string
  quantity: number
  location: string
  revisionDate: string
  packages: string[]
}

interface SdsPackage {
  id: number
  name: string
  description: string
}

Inferred from pages/Sds.tsx's use of data/mockSds.ts: a richer SdsRecord (defined locally in the mock file, not in types.ts) extends far past SdsItem — molecularFormula, flashPoint, boilingPoint, density, vaporPressure, solubility, autoIgnitionTemp, phRange, storageTemp, notes, lastUpdated, tenants: TenantRef[], with TenantRef = { id: number, name: string }.

Status codes are one shared vocabulary — the demo Table's local 4-state (Open/In Progress/Done/Cancelled) is unrelated; don't conflate them. And types.ts currently only holds SdsItem/SdsPackage — the richer shapes live scattered in data/*.ts. When rebuilding, either preserve that split or fold everything into types.ts per the project's own convention.

7 · Rebuild checklist

  1. Scaffold. Vite + React + TS; Tailwind v4 via @tailwindcss/vite; @iconify/react + @iconify-json/mdi; flag-icons; react-router-dom; mgsmu-react (or a minimal local equivalent) + react-swipeable.
  2. Global CSS. src/style/app.css — @import "tailwindcss", dark custom-variant, thin scrollbar rules, user-select: none with input/textarea override.
  3. Theme + Accent contexts. ThemeContext (localStorage app-theme, toggles .dark), AccentContext (7-color palette, localStorage app-accent, sets the two CSS vars). Wrap <App> in both.
  4. Types. Create src/types.ts with SdsItem, SdsPackage, and fold in SdsRecord/TenantRef/upload-state (better centralized than the original's split — see §5).
  5. Primitives first. All 6 Button components, then FormElements starting with TextInput, DropdownSelect, DatePicker, ToggleSwitch, SegmentedControl — the compound-modal-picker pattern generalizes from there.
  6. Status/Chip system. colorUtils.ts → Status.tsx (STATUS_MAP + 3 forms) → chip variants (Muted/Outline/Square) → generic Chip → domain chips (TenantChip, DueDateChip, UploadChip, DateTag family).
  7. Shell. splitview/Panel.tsx (resizable/collapsible primitive) → splitview/Layout.tsx (3-panel composition, route-conditional visibility, global modals/toast mount).
  8. Nav. NavItem, NavDivider, NavFavorites (+ useFavorites), NavItemModal.
  9. Modals. Modal base → ConfirmModal → domain modals as pages need them.
  10. Tables. Table.tsx (responsive/selectable) → SortableHeader → domain tables.
  11. Pages. Showcase pages (Buttons, Chips, Statuses, Form) via the SectionHeader pattern first — cheapest to verify visually — then domain pages (Sds, Tenants, Packages) composing the Sds field-editing kit.
  12. Wire routing. App.tsx table inside <Layout>; verify per-route panel-visibility conditions match §3.
  13. Polish pass. Accent-var consistency (inline style/CSS var, never a Tailwind color class), dark-mode parity on every new component, JS hover handlers on primary-variant buttons.

Gaps / not directly verified

  • Unopened files — characterized only by name/sibling pattern: InlineInput, NumberPicker, ColorPicker, TimePicker, GrowingTextarea, HintText/HintField, ImageSlider, Title family, ButtonRow, StatusChipSquare, StatusProgress, DateTag, SdsDateTag, TenantChipCompact, UploadChip, SdsTable, TenantsTable, PackagesTable, all Picker modals except Modal/ConfirmModal, Checkbox internals, TagField, QualityTags, most hooks, and 4 of 7 page bodies.
  • No tailwind.config.js — Tailwind v4 is CSS-first, so there's no central token file; the values above are reverse-engineered from literal classes and hex values in components.
  • --font-primary is referenced in app.css but never defined anywhere sampled — the actual font family is genuinely unresolved.
  • No test framework or CI present in the repo.
  • react-swipeable is a listed dependency never observed in a sampled component — likely a mobile-nav or table-row-swipe interaction that wasn't opened.
  • Animation durations transferred faithfully where explicit in Tailwind classes (duration-150/200); the download-button progress fill is confirmed to use requestAnimationFrame, not a CSS transition.
